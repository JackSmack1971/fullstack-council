# v3.0 Dual-Lane Orchestration Notes

**Conceptual Model:**
The original v3.0 design used a "Dual-Lane" approach to balance structure and speed:

1. **Strict Path (Lanes A-G)**: Hardened, 5-8 step sequential chains for critical features (Architecture -> Component -> UI -> Quality).
2. **Fast Path (Lane H / Hot Take)**: A 2-step bypass for "vibes-based" prototyping or standard T3 patterns, skipping formal architecture reviews.

**Rationale for Removal (v3.5 Refactor):**
The "Fast Path" introduced non-deterministic variability and technical debt "bypass" opportunities. In v3.5, all feature implementations are consolidated into a single, deterministic, axiom-enforced lane (`/chain-a-feature`) to ensure end-to-end type safety (T3 Axioms) and zero-rot documentation.
